# 移动端独立 UI 设计

## 目标

在不删减任何现有功能的前提下，为买家端新增一套移动端专用 UI。

要求：

- 桌面端继续保留当前工具台式布局
- 移动端进入后使用独立的工具箱首页和功能详情页交互
- 功能能力必须与桌面端完全一致
- 移动端不能只是缩小桌面端布局，而是明确做成另一套更像轻 App / 小程序的体验

## 当前已确认范围

- 按设备切换不同 UI
- 移动端首页使用单列大卡片工具箱
- 移动端详情页使用“整屏切页感”的独立功能页
- 不做视觉 Companion

## 设计结论

采用：

- `共享数据，分离视图`

也就是：

- conversion catalog、上传、校验、提交、结果下载、进度展示等逻辑继续共用
- 桌面端与移动端只在视图层和交互层分开
- 不额外拆成两个后端入口，也不复制转换逻辑

这样可以避免：

- 桌面端和移动端功能逐步漂移
- 未来新功能要改两套业务逻辑
- 维护成本失控

## 移动端首页

移动端首页改成单列工具箱：

- 每张卡片纵向占满宽度
- 只显示：
  - 功能名
  - 一句话简介
  - 进入按钮
- 首屏不直接显示上传表单
- 页面整体更短、更直白，减少信息噪音

推荐移动端首页结构：

1. 顶部品牌 / 简短标题
2. 卡密登录区
3. 登录成功后显示单列工具卡片
4. 工具卡片点击进入独立详情页

## 移动端详情页

移动端详情页采用整屏切页感：

- 点击某个工具后，隐藏工具列表
- 展示一个独立功能页
- 顶部固定：
  - 返回工具列表
  - 当前功能标题
- 中部：
  - 文件选择
  - 参数表单
  - 进度
  - 结果下载

详情页不再显示：

- 桌面端首页那种大 Hero 介绍
- 桌面端工具总览信息
- 当前任务无关的其它工具内容

## 设备判定

移动端 UI 先按前端设备条件切换：

- 以 viewport 宽度为主
- 同时允许根据 `pointer: coarse` 做辅助判断

第一版建议规则：

- `max-width` 进入移动端 UI
- 其余保持桌面端 UI

不做：

- 服务端 UA 分流
- 独立移动域名
- 两套 HTML 入口

## 组件边界

建议拆成：

- 桌面端 overview / detail 继续沿用当前结构
- 新增移动端 overview 视图
- 新增移动端 detail 视图
- 共用 catalog 数据
- 共用 conversion submit 流程
- 共用结果卡与上传进度逻辑，必要时只做移动端包装样式

需要特别保证：

- 新增功能后，桌面端和移动端都能自动出现在目录中
- 不要手写两份功能清单

## 状态流

移动端至少需要这些状态：

- 未登录
- 已登录 + 工具列表
- 已进入某个工具详情页
- 正在上传
- 正在处理
- 处理完成
- 处理失败

建议继续保留当前已有的：

- `conversionCatalog`
- `conversionSummaries`
- 详情页当前工具 key

但新增一个移动端当前路由态，例如：

- `mobileView = overview | detail`
- `mobileActiveConversionKey`

## 交互约束

移动端必须保证这些行为：

- 返回按钮回到工具列表，不丢失登录态
- 同一功能的参数逻辑与桌面端一致
- 上传和结果下载在手机上可用
- 复杂功能如：
  - PDF 加水印
  - PDF 转 Word
  - PDF 签名 / 盖章
  在移动端仍然能完成

## 风险点

1. 现有 `app.js` 已经越来越重  
需要避免把移动端分支硬塞成更多条件判断，最好提取共享逻辑与视图渲染函数。

2. 手写签名在移动端的 touch 交互  
需要专门验证触摸绘制是否顺滑，不能只看桌面鼠标行为。

3. 多文件排序类功能  
如 `PDF 合并` 在移动端如果继续保留上下移动，按钮尺寸和点击区域要专门适配。

## 测试要求

至少覆盖：

- 桌面端现有视图不回归
- 小屏进入移动端 UI
- 登录后显示单列工具卡片
- 点击工具进入移动端详情页
- 返回按钮回到工具列表
- 关键功能在移动端至少完成一次真实链路：
  - PDF 合并
  - PDF 压缩
  - PDF 转 Word
  - PDF 加水印
  - PDF 签名 / 盖章

## 范围外

这一轮不做：

- 原生 App
- PWA
- 服务端设备分流
- 移动端独立接口
- 移动端动画系统大重构
