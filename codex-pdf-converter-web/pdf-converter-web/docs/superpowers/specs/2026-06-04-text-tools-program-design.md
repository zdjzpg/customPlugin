# 文本工具总设计

## 目标

在当前买家端工具站中新增一个完整的 `文本工具` 分类，并最终覆盖 UU 文本工具页的主要能力集合。

开发策略：

- 内部按 4 个阶段顺序实现
- 对外等全部完成后再统一上线
- 每一阶段都基于同一套前端架构扩展，不重复推翻

## 总体范围

最终文本工具分为 4 组：

1. 高频文本处理
2. 提取与筛选
3. 列表与表格辅助
4. 高级与长尾工具

## 第一阶段：高频文本处理

第一阶段先实现最常用、最稳定、纯前端即可完成的文本工具：

- 文本去重
- 删除空行
- 删除所有空格
- 批量替换
- 字符数统计
- 英文大小写转换

这些工具共同特点：

- 无需后端转换
- 无需额外系统依赖
- 结果即时返回
- 能快速验证整套文本工具架构是否成立

## 后续阶段规划

### 第二阶段：提取与筛选

- 链接提取
- 邮箱提取
- 手机号提取
- 域名提取
- IP 提取
- 数字提取
- 中文提取
- 英文单词提取

### 第三阶段：列表与表格辅助

- 文本转列表
- 列表转文本
- 列表排序
- 列表随机打乱
- 列表重复统计
- 列表前后缀添加
- 列表截取
- 列表字符删除

### 第四阶段：高级与长尾工具

- UUID 生成
- 正则测试 / 正则提取
- Unicode 编解码
- 金额大写转换
- 违禁词检测
- 繁简体转换
- 中英文符号转换
- 更多长尾文本处理工具

## 前端架构

### 分类结构

买家端分类从当前：

- `PPT 工具`

扩展为：

- `PPT 工具`
- `文本工具`

### 工具来源

工具来源分成两类：

1. 服务端工具
   - 来自 `/api/conversions/catalog`
   - 主要是文件上传/转换类

2. 本地文本工具
   - 前端内置 catalog
   - 前端内置运行逻辑

### 统一模型

前端统一工具对象至少包含：

- `key`
- `label`
- `helperText`
- `kind`
- `categoryKey`

其中：

- 文件工具：`kind = conversion`
- 文本工具：`kind = local_text`

## 交互模型

### 文本工具列表页

文本工具与现有工具一样出现在卡片列表中。

点击后进入工具详情页。

### 文本工具详情页

文本工具详情页采用本地处理模式：

- 输入区
- 参数区
- 开始处理按钮
- 输出区 / 统计区

不走：

- 文件上传
- 后端转换
- 下载结果文件

### 页面状态

当前已有：

- 工具分类
- 搜索
- 详情页 URL 状态恢复

文本工具要直接复用这套状态系统。

## 设计约束

### 第一阶段不做

- 历史记录
- 收藏
- 文本工具结果导出文件
- 复杂富文本编辑
- 大规模超长文本性能专项优化

### 第一阶段必须保证

- 桌面端与移动端都可用
- 刷新后仍停留在当前文本工具详情页
- 搜索能搜到文本工具
- 文本工具不影响现有文件转换工具

## 测试要求

第一阶段至少覆盖：

- `文本工具` 分类显示
- 文本工具能进入详情页
- 文本去重逻辑
- 删除空行逻辑
- 删除所有空格逻辑
- 批量替换逻辑
- 字符数统计逻辑
- 英文大小写转换逻辑
- 文本工具详情页刷新后状态恢复
