# 编程工具总设计

## 目标

在当前买家端工具站中新增一个完整的 `编程工具` 分类，并逐步覆盖 `https://uutool.cn/type/code/` 中适合当前产品形态的主要工具能力。

对外目标：

- 首页新增一级类目 `编程工具`
- 进入类目后直接展示普通工具卡，不做组内分区
- 页面观感接近 `uutool` 的编程开发页长列表工具墙
- 搜索继续覆盖全站，能搜到 `PPT 工具`、`文本工具`、`编程工具`

开发策略：

- 三类工具最终都放在同一个 `编程工具` 类目中
- 内部按执行方式分层实现，不把全部逻辑塞进一种运行模型
- 允许按批次交付，但每一批都直接接入同一个类目，不做无功能占位卡

## 总体范围

`编程工具` 最终包含三类能力：

1. 纯前端本地工具
2. 轻后端抓取 / 解析工具
3. 网络探测 / 外部查询工具

这三类在用户界面上都是普通工具卡，用户不需要感知底层实现差异。

## 前端分类结构

买家端一级分类从当前：

- `PPT 工具`
- `文本工具`

扩展为：

- `PPT 工具`
- `文本工具`
- `编程工具`

`编程工具` 的列表页规则：

- 不再像 `文本工具` 那样按小组展示
- 直接平铺所有可用工具卡
- 桌面端和移动端都沿用当前买家端的卡片与详情页模式

## 统一工具模型

前端统一工具对象至少包含：

- `key`
- `label`
- `helperText`
- `kind`
- `categoryKey`

其中：

- 文件转换工具：`kind = conversion`
- 文本工具：`kind = local_text`
- 编程工具本地执行：`kind = local_dev_tool`
- 编程工具后端解析：`kind = backend_dev_tool`
- 编程工具网络探测：`kind = network_dev_tool`

`categoryKey` 为：

- `ppt_tools`
- `text_tools`
- `dev_tools`

这样可以保持：

- 页面呈现统一
- 搜索入口统一
- 详情页外壳统一
- 底层执行逻辑按工具类型分流

## 交互模型

### 列表页

`编程工具` 在首页和侧边导航中与其他一级类目并列。

进入后直接展示普通工具卡，每个工具卡保持当前产品的统一信息结构：

- 工具名
- 一句话说明
- `查看详情`

### 详情页

所有编程工具继续复用当前详情页框架：

- 顶部标题与简要说明
- 参数区 / 输入区
- `开始处理` 或同义操作按钮
- 结果区
- `返回列表`

根据不同执行类型切换具体表单和执行方式：

1. `local_dev_tool`
- 全部在浏览器本地处理
- 适合文本编解码、格式化、哈希、进制转换一类工具
- 结果即时显示，不依赖后端接口

2. `backend_dev_tool`
- 由前端提交文本或 URL 到你自己的后端
- 适合 `sitemap 提取`、`html 链接提取`、`网页 HTML 读取` 一类工具
- 结果返回后展示在页面内，不走文件下载为主

3. `network_dev_tool`
- 由后端发起网络探测或查询
- 适合 `SSL 检测`、`Nslookup`、`证书解析`、`主机名反查` 一类工具
- 结果以结构化信息块或文本结果形式展示

用户侧始终保持统一心智：

- `查看详情`
- 填输入
- 点击执行
- 查看结果

## 分批落地顺序

### 第一批：纯前端本地编程工具

第一批先实现不依赖后端的新工具，目标是快速把 `编程工具` 类目建立起来，并验证统一前端架构。

优先范围：

- `UUID 生成`
- `Base64 加解密`
- `中文转 Unicode`
- `Unicode 还原`
- `全角转半角`
- `半角转全角`
- `文本转十进制 Unicode`
- `md5 加密`
- `md5 批量加密`
- `字符串哈希/散列`
- `JSON 格式化`
- `JSON 编辑器`
- `HTML 实体编解码`
- `HTML 转 JS`
- `HTTP 头转 JSON`
- `URL 编解码`
- `2-36 任意进制转换`
- `文本转二进制`
- `二进制转文本`
- `文本转进制`
- `进制转文本`
- `时间戳转换`
- `Crontab 解析`
- `Cookie 转 JSON`
- `JSON 字段提取`
- `列表转 JSON`
- `JSON 转列表`
- `JSON 转 PHP`
- `JS 对象转 JSON`
- `JSON 转 JS 对象`
- `RSA 密钥对生成`
- `JS 美化压缩`
- `CSS 美化压缩`
- `HTML 美化压缩`

这些工具特点：

- 不需要文件上传
- 不依赖服务端网络访问
- 最适合沿用当前 `文本工具` 的本地执行模式扩展

### 第二批：轻后端抓取 / 解析工具

第二批实现由你自己的服务端抓取或解析的工具。

优先范围：

- `sitemap 链接提取`
- `html 链接提取`
- `网页链接提取`
- `网页 HTML 读取`
- `URL 转 sitemap`
- `Robots 生成`
- `批量添加 http/https 协议`
- `HTML 预览`

这批特点：

- 不直接依赖特定第三方中文数据接口
- 结果主要是文本、链接列表或结构化字符串
- 能和当前 Node 后端天然结合

### 第三批：网络探测 / 外部查询工具

第三批再做需要后端主动联网探测的工具。

优先范围：

- `网站 SSL 证书检测`
- `SSL 证书解析`
- `SSL 证书过期查询`
- `Nslookup 查询`
- `IP 地址获取主机名`
- `浏览器 UA 查询`
- `JS 设备屏幕参数检测`
- `浏览器指纹检测`

放在第三批后半段再评估的范围：

- `ICP备案查询`
- `ICP备案批量查询`
- `ICP备案反查`
- `IP 归属地`
- `多节点 IP 检测`

原因：

- 这些功能更依赖外部数据源、额外采集链路或稳定的网络环境
- 失败原因比前两批复杂
- 更需要提前定义超时、频率限制和结果可信度边界

## 技术边界

### 前端

前端保持一个统一买家端壳子，不额外创建第二套产品页面。

需要扩展的只是：

- 新类目元数据
- 新工具 catalog
- 按 `kind` 分流详情页渲染
- 按 `kind` 分流执行逻辑

### 后端

后端不把所有编程工具塞进现有文件转换接口。

建议新增与编程工具独立的处理入口，例如单独的 API 路由层，用来处理：

- 页面抓取
- 文本解析
- SSL / DNS / 网络检测

这样可以避免：

- 混淆文件转换与文本 / 网络工具的职责
- 在现有转换统计和文件上传逻辑上硬加分支

### 使用统计

当前管理端统计规则应延续：

- 只有用户真正点击执行动作时才记一次使用
- 进入详情页不计数

但需要确保新增编程工具也有独立中文名称映射，避免后台统计出现内部 key。

## 错误处理

三类工具分别做边界处理：

### 本地工具

- 非法 JSON
- 非法正则
- 非法进制
- 空输入但工具要求有输入

要直接在前端给出明确提示，不把错误信息写成技术栈术语。

### 后端解析工具

- 抓取超时
- 目标 URL 无法访问
- 返回内容为空
- 内容过大
- 非预期内容类型

结果区要给用户可理解的失败原因，例如“目标网页暂时无法读取，请确认链接可直接访问”。

### 网络探测工具

- DNS 解析失败
- SSL 握手失败
- 连接超时
- 目标拒绝连接
- 查询无结果

不能把底层异常堆栈直接暴露给买家端。

## 设计约束

当前阶段不做：

- 编程工具收藏
- 历史查询列表
- 批量任务队列
- 大规模异步任务面板
- 高级开发者工作台式布局
- 在 `编程工具` 内再二次分组展示

当前阶段必须保证：

- `编程工具` 能作为一级类目正常显示
- 类目内工具卡直接平铺
- 搜索能搜到编程工具
- 桌面端与移动端都能进入详情页并执行
- 本地工具、后端工具、网络工具不会互相污染执行逻辑
- 现有 `PPT 工具` 和 `文本工具` 行为不回归

## 测试要求

至少覆盖以下层面：

1. 分类与路由
- `编程工具` 一级类目显示
- 列表可进入详情页
- 刷新后仍停留在当前编程工具详情页

2. 本地工具运行时
- 至少覆盖首批几个代表性工具：
  - `UUID 生成`
  - `Base64 加解密`
  - `JSON 格式化`
  - `md5 加密`
  - `任意进制转换`

3. 后端解析接口
- 成功抓取
- URL 无效
- 超时
- 空结果

4. 网络探测接口
- 成功返回证书 / DNS 信息
- 握手失败或域名无效时返回可理解错误

5. 浏览器真实回归
- `编程工具` 类目卡显示
- 搜索命中编程工具
- 一个本地编程工具完整可用
- 一个后端解析工具完整可用
- 一个网络探测工具完整可用
